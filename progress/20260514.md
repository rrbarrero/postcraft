# Progress - 2026-05-14

## First Self-Evals

After adding the first self-evals for the local draft-generation pipeline, the
test suite exposed several quality problems in the current Phase 1 and Phase 2
flow.

These failures are useful because they confirm that the evals are now checking
real product behavior instead of only validating helper functions. The current
pipeline can create a workspace, but the generated artifacts still do not meet
the editorial and grounding contract we want for PostCraft.

## What These Failures Mean

### `draft.md` Is Not A Standalone Article

The draft still looks too much like an intermediate pipeline artifact. It
contains structure from the outline and analysis phases instead of reading as a
finished technical article.

This breaks the core writing contract: the writer should return only the
Markdown draft, not the planning material used to produce it.

### Pipeline Headings Leak Into The Draft

The draft contains forbidden headings such as:

- `## Outline`
- `Technical Analysis`

This means the writer is copying or preserving upstream artifact structure
instead of transforming it into article prose.

### The Draft Does Not Have Enough Prose

The eval expects at least three prose paragraphs. The generated draft does not
meet that threshold, which suggests that the output is still too schematic,
bullet-heavy, or outline-shaped.

### Generic Language Still Appears

The draft contains the phrase:

```text
scalable
```

This is exactly the kind of generic portfolio language PostCraft should avoid
unless the repository provides concrete evidence for it.

### The Technical Review Does Not Reference File Paths

The technical review currently does not cite any file paths. That makes the
review harder to trust because claims are not tied back to repository evidence.

For the reviewer to be useful, supported and unsupported claims should point to
files such as:

- `README.md`
- `Cargo.toml`
- `src/cli.rs`
- `src/main.rs`
- representative domain, application, or infrastructure files

## Problems To Solve Next

1. Strengthen the writer prompt so `draft.md` is a standalone article and never
   includes planning sections such as `## Outline` or `Technical Analysis`.
2. Add a deterministic artifact validator before the technical review so
   malformed drafts can be caught immediately.
3. Improve the outline-to-draft transition so the writer uses the outline as a
   plan, not as content to paste.
4. Tighten generic-language rules in the writer prompt, especially for words
   like `scalable`, `production-ready`, `full-featured`, and similar claims.
5. Update the technical reviewer prompt so every important supported or
   unsupported claim includes file-path evidence when available.
6. Improve `project_facts.md` and `technical_analysis.md` so downstream agents
   receive clearer file-backed facts.

## Status

The evals are doing their job: they are exposing the gap between "the pipeline
runs" and "the generated post is good enough to be used as portfolio content."

The next development step should focus on making the generated artifacts pass
these self-evals before adding more publishing-oriented functionality.
